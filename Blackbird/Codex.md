BlackBird launch prediction redesign context

Project:
BlackBird is a KSP rendezvous launch planning/guidance addon.

Hard Rules - DO NOT VIOLATE

- No temporary code.
- No placeholder implementations.
- No "we'll replace this later" architecture.
- If a class would eventually be deleted, do not create it.
- Agree on contracts before implementation.
- Implement one file at a time.
- LaunchPlanner is the mission planner.
- AscentGuidance is the flight control computer.
- AttitudeControl is the actuator layer.
- LaunchPlan is the single source of truth for the mission plan.
- VesselState is a current vessel snapshot only.
- Prefer adding reusable math to OrbitMath.
- Prefer extending existing files over creating new files unless the responsibility is clearly separate.
- Keep MVP lean; flavor/debug metrics can be added later.

Moonshot Goal

User selects a target vessel already in orbit.

BlackBird computes:
- launch time
- insertion orbit
- ascent profile
- rendezvous strategy

User chooses a candidate.

BlackBird automatically guides or flies the ascent.

Result should be a near-rendezvous with minimal correction burns.

The planner should know the intended rendezvous strategy before launch, not invent it after reaching orbit.

Current working state:
- Warp system fixed
- FlyByWire migration complete
- SAS steering removed
- MechJeb-derived AttitudeControl implemented
- Guidance mode controls vessel attitude correctly
- Current path:
  LaunchHandler -> AscentGuidance -> AscentGuidanceInfo -> OnFlyByWire -> AttitudeControl -> FlightCtrlState


Development Goal:
Build a production ascent guidance prediction model that produces:
- selectable launch candidates
- recommended launch time
- recommended insertion apoapsis/periapsis
- recommended heading
- generated pitch/heading ascent profile
- estimated post-launch orbits to rendezvous
- estimated dV used / remaining
- real-time guidance display
- UI integration with a Choose button per candidate

Major architecture decision:
LaunchPlan remains the single source of truth for the complete flight plan.
VesselState is only a current vessel snapshot and should not be stored as duplicated planning state.

Candidate workflow:
LaunchPlanner.Create()
  -> generate LaunchCandidate[]
  -> sort valid candidates by lowest estimated dV used
  -> default SelectedCandidateIndex = 0

User can then press "Choose" on any candidate:
  -> LaunchPlan.SelectedCandidateIndex = chosen index
  -> LaunchHandler.SetPlan(plan)
  -> guidance/warp/use selected candidate

No duplicated selected fields if possible.
LaunchPlan summary values should resolve from SelectedCandidate, not copied from candidate 0.

LaunchPlanner.cs contract:
Purpose:
Generate the complete LaunchPlan, including selectable launch candidates.

Inputs:
  Vessel activeVessel
  Vessel targetVessel
  InsertionTarget insertionTarget
  LaunchLocation launchLocation

Output:
  LaunchPlan with LaunchCandidate[]

Pipeline:
1. Build VesselState from active vessel
2. Read active/target OrbitInfo
3. Build base LaunchPlan facts
4. Generate launch-window opportunities
5. Generate insertion/phasing orbit candidates
6. Build AscentProfile for each candidate
7. Estimate dV, phase error, plane error, rendezvous time
8. Create LaunchCandidate[]
9. Sort candidates:
   - valid first
   - lowest EstimatedDeltaVUsed
   - lowest error
   - shortest rendezvous time
   - earliest launch
10. Set SelectedCandidateIndex = 0
11. UI/user may later change SelectedCandidateIndex; selected candidate becomes active plan

Important:
LaunchPlanner owns candidate generation and scoring.
It does not own real-time guidance.
Reusable math goes to OrbitMath.

PhasingRecommendation:
Keep it.
It is not post-orbit advice.
It is part of prelaunch mission planning.
Meaning:
"this is the intended insertion/phasing orbit selected before launch"
not:
"once you're in orbit, maybe do this"

NASA-style planning assumption:
We should know before launch what orbit/altitude we intend to reach to get near docking region.

AscentGuidance.cs contract:
Purpose:
Flight control computer.

Consumes:
  VesselState
  LaunchPlan.SelectedCandidate
  LaunchPlan.SelectedCandidate.AscentProfile
  manual pitch/heading commands
  GuidanceMode

Outputs:
  AscentGuidanceInfo

Does not own:
- pitch profiles
- launch windows
- candidate scoring
- phasing logic
- target prediction

Behavior:
Autopilot:
  command pitch = selectedCandidate.AscentProfile.GetPitchAtAltitude(vesselState.AltitudeMeters)
  command heading = selectedCandidate.AscentProfile.GetHeadingAtAltitude(vesselState.AltitudeMeters)

Guidance:
  command pitch = manualPitchCommandDeg
  command heading = manualHeadingCommandDeg

None:
  command pitch = current vessel pitch
  command heading = current vessel heading

Remove from current AscentGuidance:
- RssPitchProfile
- StockPitchProfile
- GetTargetPitchDeg
- InterpolatePitchProfile if moved into AscentProfile
- TargetAzimuthDeg/TargetPitchDeg dependencies
